Files info
----------
This is where EPKL keeps non-layout files it needs. Unless you're an advanced user, you may not need to change anything in these files.

On the other hand, looking inside some of them should be useful if you want to understand more about how EPKL works!
<br><br>

Overview over the EPKL Prefix-Entry and other advanced syntax for mappings, useable in several contexts:
```
;;  ====================================================================================================================
;;  EPKL prefix-entry syntax is useable in layout state mappings, Extend, Compose, PowerString and dead key entries.
;;  - There are two equivalent prefixes for each entry type: One easy-to-type ASCII, one from the eD Shift+AltGr layer.
;;      ‚Üí  |  %  : Send a literal string/ligature by the SendInput {Text} method
;;      ¬ß  |  $  : Send a literal string/ligature by the SendMessage method
;;      Œ±  |  *  : Send ‚Äπentry‚Ä∫ as AHK syntax in which !+^# are modifiers, and {} contain key names
;;      Œ≤  |  =  : Send {Blind}‚Äπentry‚Ä∫, keeping the current modifier state
;;      ‚Ä†  |  ~  : Send the hex Unicode point U+<entry> (normally but not necessarily 4-digit)
;;      √ê  |  @  : Send the current layout's dead key named ‚Äπentry‚Ä∫ (often a 3-character code)
;;      ¬∂  |  &  : Send the current layout's powerstring named ‚Äπentry‚Ä∫; some are abbreviations like &Esc, &Tab‚Ä¶
;;  - Other advanced state mappings:
;;      ¬Æ¬Æ |  ¬Æ# : Repeat the previous character. `#` may be a hex number. Nice for avoiding same-finger bigrams.
;;      ¬©‚Äπname‚Ä∫  : Named Compose key, replacing the last written character sequence with something else.
;;      ##       : Send the active system layout's Virtual Key code. Good for OS shortcuts, but EPKL can't see it.
;;  ====================================================================================================================
```
